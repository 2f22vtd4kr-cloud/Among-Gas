# AMONG US TELEGRAM SPEC (MODULE 1: NETWORK ARCHITECTURE & PACKET SERIALIZATION)
## 1. Network Topology & Environment Constraints
Because the client application executes within a sandboxed Telegram Mini App Webview, it operates under web browser security constraints:
 * **No Direct UDP Sockets:** Native UDP (dgram) is unsupported. The network layer must use **Secure WebSockets (wss://)** running over TCP.
 * **No Double Reliability Layer:** Do not implement application-level packet delivery confirmations (ACKs, retries, or nonces). TCP implicitly handles packet ordering and guarantees delivery.
 * **Binary Serialization Mandatory:** Webviews experience performance degradation when parsing high-frequency text strings. The connection data type must be forced to raw binary buffers (arraybuffer on client, Buffer on server).
## 2. Secure Telegram Handshake & Verification Phase
Every incoming client connection must validate its payload against Telegram's security infrastructure before getting assigned an internal slot ID.
### Connection Lifecycle
 1. The client connects via wss://your-replit-domain.repl.co?initData=ENCODED_STRING.
 2. The server halts any game interaction, extracts the initData parameter, and performs an HMAC-SHA256 integrity token validation pass against the active bot secret key.
 3. If validation succeeds, the connection is upgraded, the user's authentic identity is mapped, and the server transmits a **Handshake Acknowledge Packet (0x01)**.
 4. If validation fails, the server transmits a **Handshake Terminate Packet (0x00)** and destroys the socket immediately.
## 3. The Complete Binary Protocol Blueprint (The Opcode Matrix)
Every frame sent across the WebSocket connection begins with a single **1-byte message header (Opcode)**, followed by an immutable, packed Little-Endian payload.
| Opcode (Byte 0) | Packet Name | Data Direction | Byte Size | Payload Layout Description |
|---|---|---|---|---|
| **0x00** | Handshake Fail | Server \rightarrow Client | 1 Byte | No payload. Instantly closes connection. |
| **0x01** | Handshake Success | Server \rightarrow Client | 2 Bytes | Byte 1: Assigned Player ID Slot (0-14). |
| **0x10** | Join/Create Lobby | Client \rightarrow Server | Variable | Byte 1: Create Flag (0=Join, 1=Create). Bytes 2-7: 6-char Room Code string. |
| **0x11** | Player Move Request | Client \rightarrow Server | 5 Bytes | Bytes 1-2: Target X (\text{Float} \times 100). Bytes 3-4: Target Y (\text{Float} \times 100). |
| **0x12** | Game Start Command | Client \rightarrow Server | 1 Byte | Sent by Host only. Triggers role shuffle execution loop. |
| **0x15** | RPC Component Event | Bidirectional | Variable | Byte 1: RPC Sub-action ID (Kill, Vent, Sabotage, Use). |
| **0xFF** | State Delta Broadcast | Server \rightarrow Client | Variable | Byte 1: Total count of active objects. Sub-blocks: Packed data arrays. |

Multi-byte numbers must be written and read using **Little-Endian (LE)**. Floating-point values (like coordinates) must be converted into 2-byte signed integers (Int16) to conserve bandwidth.
